Lepton-EDA Bridge User Guide

pqwave integrates with lepton-eda through a netlist-based file-watching pipeline and a companion Guile Scheme server that enables bidirectional cross-probe, back-annotation, and in-schematic menus. This guide covers every feature and walks through real analysis workflows.

No source modifications required. Everything works with a stock lepton-eda installation. A single companion Scheme script (pqwave-server.scm) is installed once into the lepton-eda autoload directory — it is loaded automatically by lepton-schematic on startup.

Part 1 — Feature Reference

Overview: How It Works

lepton-schematic pqwave ---------------- ------ User opens a .sch file File watcher detects save (mtime poll) User edits & saves (Ctrl+S) ───────► lepton-netlist -g spice-sdb exports clean SPICE — zero fixes needed ngspice -b -r result.raw circuit.cir Loads and displays ALL signals In-schematic menus (optional): Reverse cross-probe: Netlist > SPICE ─────────────────► Click net in schematic ───────► Simulation > ngspice ────────────► trace plotted in pqwave Wave View > pqwave ──────────────► Forward cross-probe: Bidirectional highlight via TCP :9424 Select trace in pqwave ─────────► net highlighted in schematic

Design note — why probing starts from pqwave, not the schematic: pqwave displays every signal from the simulation at once in the signal browser. The user explores freely, plotting multiple traces and using cursors to measure. The schematic is the design canvas; pqwave is the analysis cockpit. Cross-probe lets you navigate between the two in both directions on demand.

Key features of the lepton-eda bridge:

  1. Netlist pipeline — automatic export via lepton-netlist -g spice-sdb, which produces clean SPICE requiring no post-processing (unlike KiCad).
  2. Bidirectional cross-probe — highlight nets in either direction using a TCP channel (port 9424) between pqwave and the companion Scheme server running inside lepton-schematic.
  3. DC back-annotation — stamp simulated operating point voltages directly onto the schematic as netname attributes or floating text labels.
  4. In-schematic menus — Netlist, Simulation, and Wave View actions available directly from lepton-schematic's menu bar, enabling the complete simulation workflow without leaving the schematic editor.
  5. Pure Scheme plugin — no patches or recompilation of lepton-eda required. All cross-probe and menu logic is implemented in a single pqwave-server.scm file that lepton-schematic loads at startup via its autoload mechanism.

Netlist Pipeline

The bridge processes schematics through a two-stage pipeline:

StageActionDetails
Export lepton-netlist -g spice-sdb -o - file.sch Generates SPICE netlist from .sch file using the spice-sdb backend. Output is written to stdout.
Simulate ngspice -b -r output.raw circuit.cir Runs batch simulation, produces a standard .raw file that pqwave loads directly.

Why zero fixes are needed: lepton-eda's spice-sdb backend is purpose-built for SPICE netlisting. It understands SPICE device semantics and produces correct output by default. This contrasts with KiCad, whose SPICE exporter is layered on top of a general-purpose netlist generator and requires post-processing to fix node name slashes, semiconductor pin ordering, and .control block placement.

Configuration — set via Settings > External Converter Paths or the Session API:

SettingDefaultDescription
lepton_netlist Auto-detect from $PATH Path to lepton-netlist binary
ngspice Auto-detect from $PATH Path to ngspice binary
port 9424 TCP port for cross-probe communication
auto_simulate true Run simulation automatically when schematic is saved

In-Schematic Menus

Once pqwave-server.scm is installed, lepton-schematic gains three new top-level menus. These allow you to run the complete simulation workflow without leaving the schematic editor:

MenuActionWhat It Does
Netlist SPICE Runs lepton-netlist -g spice-sdb -o <base>.cir on the active schematic. Produces a .cir file in the same directory.
Simulation ngspice Generates the .cir file, then runs ngspice -b -r <base>.raw <base>.cir. Produces a .raw file for pqwave.
Wave View pqwave Launches pqwave <base>.raw — assumes the .cir and .raw files already exist in the same directory as the schematic.

Menu registration is automatic. The menus appear as soon as lepton-schematic starts with pqwave-server.scm in the autoload directory. They do not require any manual configuration or menu editing. The registration happens via add-menu calls in the script, which lepton-schematic's gafrc mechanism executes before the main menu is built.

[SCREENSHOT: lepton-schematic menu bar showing Netlist, Simulation, and Wave View menus with SPICE, ngspice, and pqwave actions visible]

Complete workflow from lepton-schematic:

  1. Open your .sch schematic in lepton-schematic.
  2. Netlist > SPICE — generates the .cir file.
  3. Simulation > ngspice — runs the simulation and produces the .raw file.
  4. Wave View > pqwave — launches pqwave with the .raw results.

Bidirectional Cross-Probe

Cross-probe lets you navigate between pqwave's analysis views and the schematic in lepton-schematic. Communication happens over a TCP channel on port 9424 between pqwave and pqwave-server.scm.

pqwave to schematic (Forward)

Select a trace in pqwave, then navigate to File > Lepton Bridge > Cross-Probe > Probe Selected Net. The corresponding net is selected and zoomed-to in lepton-schematic.

ActionMenu PathTCP Command
Probe net File > Lepton Bridge > Cross-Probe > Probe Selected Net $NET: "netname"
Probe component (via API: lepton_probe_part) $PART: "refdes"
Clear highlight File > Lepton Bridge > Cross-Probe > Clear Highlight $CLEAR

schematic to pqwave (Reverse)

When a net is selected in lepton-schematic (clicked), the Scheme server sends a $SELECTED:net message to pqwave via the TCP channel. pqwave receives this and can automatically plot the corresponding trace.

Debounce: Cursor-driven cross-probe is debounced at 250ms to avoid flooding the TCP channel when scrubbing through waveform data. Direct probe (menu click) is sent immediately.

Cursor-Driven Cross-Probe

When you track through a waveform with the X cursor in pqwave, the bridge can continuously update the schematic highlight to follow the active trace. This is controlled by the 250ms debounce timer — as you drag the cursor, the most recent position is sent after a quarter-second pause.

Connection lifecycle: The TCP connection is established when the bridge is activated (File > Lepton Bridge > Watch Schematic). It persists for the duration of the watch session. Disconnecting (Stop Watching) closes the TCP connection and removes the control bar.

Back-Annotation

The bridge can write simulation results back into the schematic. Two types of annotation are supported:

1. DC Bias Stamps

Click Annotate DC in the Lepton control bar (or navigate to File > Lepton Bridge > Annotate DC). pqwave extracts the mean voltage of every v(netname) signal from the current dataset and stamps each as a suffix on the corresponding netname attribute:

Before: netname = "VDD" After: netname = "VDD [DC:5.0 V]"

Persistence: DC stamps modify the netname attribute text. If you save the schematic after annotating, the stamps persist in the .sch file. Use Clear Annotations to strip the [DC:...] suffix from all netname attributes, restoring the clean net names.

2. Floating Labels

Voltage labels can be placed as temporary text objects near net segments using the $ANNOTATE:LABEL command (available via the API). These labels are purely visual and are removed when lepton-schematic closes or when Clear Annotations is invoked.

Persistence behavior: DC stamps modify netname attributes and will persist in the .sch file if saved. Floating labels are held in memory by the Scheme server and are lost when lepton-schematic exits. Use Clear Annotations before saving if you want to keep the schematic clean.

Session API Commands

All bridge operations are available through the Session API for scripting and headless use.

Scheme Server Installation

What is pqwave-server.scm?

pqwave-server.scm is a Guile Scheme script bundled with pqwave that provides three capabilities inside lepton-schematic:

  1. TCP server on port 9424 — receives cross-probe and back-annotation commands from pqwave.
  2. Reverse cross-probe — hooks into lepton-schematic's select-objects-hook to send $SELECTED:net events back to pqwave when the user clicks a net.
  3. In-schematic menus — registers Netlist > SPICE, Simulation > ngspice, and Wave View > pqwave menu actions.

Installation

The script lives at:

pqwave/bridge/lepton/pqwave-server.scm          (bundled with pqwave)

It must be copied to the lepton-eda autoload directory:

~/.config/lepton-eda/scheme/autoload/pqwave-server.scm    (installed)

GUI installation: When you first use File > Lepton Bridge > Watch Schematic, pqwave checks whether the script is installed. If not (or if the bundled version is newer), a dialog asks whether to install or update it. Click Yes, and pqwave copies the file. Restart lepton-schematic for the changes to take effect.

Headless installation:

api.lepton_config("install_server")

This copies pqwave-server.scm to the autoload directory without a GUI prompt. Returns {"status": "ok", "target_path": "..."} on success or {"status": "error", "message": "..."} on failure.

Update Detection

pqwave compares the modification time of the bundled script against the installed copy. If the bundled version is newer (e.g., after upgrading pqwave), it offers to update. This ensures the Scheme server stays in sync with new bridge features.

How It Loads

lepton-schematic loads all .scm files in the autoload directory at startup via the system-gafrc mechanism. No manual configuration is needed — once the file is present, lepton-schematic automatically:

  1. Loads the script and starts the TCP server on port 9424.
  2. Registers the Netlist, Simulation, and Wave View menus.
  3. Installs the select-objects-hook for reverse cross-probe.

Port check: To verify the server is running, check that lepton-schematic is listening on port 9424:

ss -tlnp | grep 9424
# or
lsof -i :9424

If the port is not listening, verify the file is installed and restart lepton-schematic.

Part 2 — Worked Tutorials

Tutorial 1: TwoStageAmp — Complete Workflow

Follow the full simulation pipeline from a lepton-eda schematic through to waveform analysis in pqwave.

Prerequisites:

  • lepton-eda installed with lepton-netlist in PATH
  • ngspice 42+ in PATH
  • pqwave installed
  • A TwoStageAmp.sch schematic (or any SPICE-ready schematic)

Step 1: Open the Schematic

  1. Launch lepton-schematic: lepton-schematic TwoStageAmp.sch.
  2. Verify the circuit: a two-stage BJT amplifier with V1 (input), VCC (supply), Q1/Q2 (transistors), R1–R6 (biasing and load), and C1–C3 (coupling capacitors).
  3. Confirm SPICE analysis directives are present on the schematic: .tran 0.1u 5m and .model statements.
[SCREENSHOT: TwoStageAmp.sch in lepton-schematic showing the complete circuit with transistors, resistors, capacitors, and SPICE directives]

Step 2: Generate the Netlist

  1. In lepton-schematic, select Netlist > SPICE.
  2. A TwoStageAmp.cir file is created in the schematic's directory.
  3. The status bar confirms: "SPICE netlist written: TwoStageAmp.cir".

Or via command line:

lepton-netlist -g spice-sdb -o TwoStageAmp.cir TwoStageAmp.sch

Step 3: Run the Simulation

  1. In lepton-schematic, select Simulation > ngspice.
  2. This regenerates the .cir and runs ngspice in batch mode.
  3. A TwoStageAmp.raw file is produced.

Or via command line:

ngspice -b -r TwoStageAmp.raw TwoStageAmp.cir

Step 4: View Results in pqwave

  1. In lepton-schematic, select Wave View > pqwave.
  2. Alternatively, in pqwave: File > Lepton Bridge > Watch Schematic, select TwoStageAmp.sch.
  3. The bridge runs the pipeline and loads the .raw results. The signal browser populates with all available signals.

Step 5: Browse and Analyze Signals

  1. Double-click v(in) — the 1 kHz input sine wave.
  2. Double-click v(stage1) — the output of the first common-emitter stage (inverted, amplified).
  3. Double-click v(out) — the final output through the coupling capacitor (DC offset removed).
[SCREENSHOT: pqwave showing three traces — v(in), v(stage1), v(out) — plotted on a shared time axis. v(in) is a small sine, v(stage1) is inverted and ~10x larger, v(out) is inverted and ~100x larger.]

Step 6: Measure Gain with Cursors

  1. Enable Cursor A (press A) and place it on v(out) at a peak of the sine wave.
  2. Enable Cursor B (press B) and place it on v(in) at the corresponding peak.
  3. The status bar shows the voltages at both cursor positions.
  4. Gain = v(out) at cursor A / v(in) at cursor B.

Expected results:

  • Stage 1 gain: approximately 10x (20 dB) from the common-emitter configuration.
  • Stage 2 gain: similar, for a total gain of approximately 100x (40 dB).
  • v(out) is AC-coupled (no DC offset) through the output capacitor.

Auto-simulate: After the initial watch, you can edit the schematic (e.g., change a resistor value) and press Ctrl+S to save. pqwave detects the file change (via 1-second mtime polling), re-runs the pipeline, and reloads the results automatically. The control bar shows the watch and simulation status.

Full workflow via Session API:

import pqwave.session as session

api = session.SessionAPI()

# Watch and simulate
api.lepton_watch("/home/user/projects/TwoStageAmp.sch")
api.lepton_simulate("/home/user/projects/TwoStageAmp.sch")

# Plot key signals
api.select_panel(1)
api.add_trace("v(in)")
api.add_trace("v(stage1)")
api.add_trace("v(out)")

# Measure gain
gain_db = api.eval_expr("20*log10(abs(max(v(out))/max(v(in))))", panel=1)
print(f"Voltage gain: {gain_db:.1f} dB")

Tutorial 2: Bidirectional Cross-Probe

This tutorial demonstrates navigation between pqwave's waveform display and lepton-schematic's circuit view in both directions.

Prerequisites:

  • pqwave-server.scm installed in the autoload directory
  • lepton-schematic running with a schematic open
  • pqwave watching the same schematic
  • Simulation completed (results loaded)

Direction 1: pqwave to Schematic (Forward Cross-Probe)

  1. In pqwave, double-click v(out) in the signal browser to plot it.
  2. Click on the trace to select it (it becomes highlighted).
  3. Navigate to File > Lepton Bridge > Cross-Probe > Probe Selected Net.
  4. In lepton-schematic, the net named out is selected and the view zooms to show it.
[SCREENSHOT: Side-by-side — pqwave on the left showing v(out) trace selected, lepton-schematic on the right showing the "out" net highlighted in the selection color]

Direction 2: Schematic to pqwave (Reverse Cross-Probe)

  1. In lepton-schematic, click on the VCC net (the power supply rail).
  2. The Scheme server sends $SELECTED:net VCC to pqwave over the TCP channel.
  3. pqwave's chat panel logs: [lepton] Selected net: VCC.

API-driven reverse cross-probe handling:

# When a net is clicked in lepton-schematic, you can script the response:
# The main_window handler (_on_lepton_net_selected) receives the net name.
# Use the API to programmatically plot it:

def on_net_selected(netname):
    signal_name = f"v({netname})"
    api.add_trace(signal_name)

Cursor-Driven Cross-Probe

  1. Select a trace in pqwave.
  2. Drag Cursor XA along the waveform.
  3. After a 250ms debounce pause, pqwave sends the net highlight to lepton-schematic.
  4. As the cursor moves through different parts of the waveform, the corresponding net stays highlighted.

Tutorial 3: DC Back-Annotation

Back-annotation lets you verify DC operating points directly on the schematic. This is useful for checking bias voltages, identifying deviation from the expected design, and documenting the operating point for review.

Step 1: Run the Simulation

  1. Watch and simulate a schematic (as in Tutorial 1).
  2. Ensure the simulation includes a .op or .tran analysis that produces DC operating point data.

Step 2: Annotate DC Voltages

  1. In pqwave's Lepton control bar, click Annotate DC (or navigate to File > Lepton Bridge > Annotate DC).
  2. pqwave extracts the mean value of every v(netname) signal and sends $ANNOTATE:DC commands to lepton-schematic.

What happens in the schematic:

Net: "VCC" → "VCC [DC:12.0 V]" Net: "Vbias" → "Vbias [DC:1.65 V]" Net: "in" → "in [DC:0.0 V]" Net: "stage1" → "stage1 [DC:6.2 V]" Net: "out" → "out [DC:6.0 V]"
[SCREENSHOT: lepton-schematic showing DC voltage stamps appended to netname attributes across the TwoStageAmp schematic. Each net label displays its simulated DC voltage in brackets.]

Step 3: Verify Against Design

  1. Check the bias point of Q1: V(stage1) should be approximately VCC/2 (6 V for a 12 V supply). A value of 6.2 V indicates good biasing.
  2. Check V(in) = 0.0 V — the input is AC-coupled, so the DC level at the base should be 0 V relative to ground.
  3. If any voltage deviates significantly, adjust the corresponding resistor value, re-simulate, and re-annotate.

Step 4: Clear Annotations

  1. Click Clear Annotations in the Lepton control bar (or navigate to File > Lepton Bridge > Clear Annotations).
  2. All [DC:... V] suffixes are stripped from netname attributes.
  3. All floating text labels are removed from the page.
  4. The schematic returns to its clean, pre-annotation state.

Important: If you save the .sch file while DC stamps are present, the [DC:... V] suffixes will be written into the file and will persist across sessions. Always use Clear Annotations before saving if you want to keep the schematic clean. Floating labels are in-memory only and are always lost when lepton-schematic closes.

API example:

# Annotate with auto-extracted voltages
api.lepton_annotate_dc()

# Annotate with explicit voltages
api.lepton_annotate_dc({
    "VCC": 12.0,
    "Vbias": 1.65,
    "in": 0.0,
    "stage1": 6.2,
    "out": 6.0,
})

# Clear everything
api.lepton_clear_annotations()

Standalone Scripting Example

A complete Python script that integrates lepton-eda schematic editing with pqwave simulation and analysis — all from a headless script: 

import pqwave.session as session

api = session.SessionAPI()

# Step 1: Install the Scheme server (first time only)
result = api.lepton_config("install_server")
if result["status"] == "ok":
    print(f"Server installed to {result['target_path']}")
    print("Restart lepton-schematic for the changes to take effect.")

# Step 2: Configure the bridge
api.lepton_config("port", 9424)
api.lepton_config("auto_simulate", True)

# Step 3: Watch and simulate
sch_path = "/home/user/projects/TwoStageAmp.sch"
api.lepton_watch(sch_path)
result = api.lepton_simulate(sch_path)

print(f"Return code: {result['returncode']}")
print(f"Raw file: {result['raw_file']}")

# Step 4: Plot key signals
api.select_panel(1)
api.add_trace("v(in)")
api.add_trace("v(stage1)")
api.add_trace("v(out)")

# Step 5: Back-annotate DC operating point to the schematic
api.lepton_annotate_dc()

# Step 6: Cross-probe the output net
api.lepton_probe_net("out")

# Step 7: Measure stage gains
gain1 = api.eval_expr("max(v(stage1)) / max(v(in))", panel=1)
gain2 = api.eval_expr("max(v(out)) / max(v(stage1))", panel=1)
print(f"Stage 1 gain: {gain1:.2f} ({20*gain1.log10():.1f} dB)")
print(f"Stage 2 gain: {gain2:.2f} ({20*gain2.log10():.1f} dB)")

# Step 8: Clean up — clear annotations before saving
api.lepton_clear_annotations()

# The file watcher keeps running in the background.
# Save the schematic in lepton-schematic to trigger automatic
# re-simulation. Press Ctrl+C in the REPL to stop.

Expected output:

Return code: 0
Raw file: /tmp/pqwave_lepton_XXXXXX.raw
Stage 1 gain: 9.87 (19.9 dB)
Stage 2 gain: 12.34 (21.8 dB)

Part 3 — Troubleshooting

lepton-netlist not found

Symptom: FileNotFoundError: lepton-netlist not found.

Solution:

  1. Verify lepton-eda is installed: 
    which lepton-netlist
  2. If not found, install lepton-eda: 
    # Debian/Ubuntu
sudo apt install lepton-eda

# Or build from source:
git clone https://github.com/lepton-eda/lepton-eda.git
cd lepton-eda && mkdir build && cd build
cmake .. && make && sudo make install
  3. If installed but not in PATH, set the path in pqwave: Settings > External Converter Paths > lepton-netlist and enter the full path (e.g., /usr/local/bin/lepton-netlist).

ngspice simulation errors

Symptom: Simulation fails with a non-zero return code.

Solution:

  1. Check the ngspice error output in pqwave's chat panel. Common causes:
    • Missing model: The .sch references a .model that is not embedded in the file. Add the model to the schematic as a SPICE directive block.
    • Floating node: A net has no DC path to ground. Add a large-value resistor (e.g., 1G) to ground.
    • Timestep too small: The transient analysis requires a smaller timestep than ngspice can handle. Adjust .tran parameters.
  2. Manually run the pipeline to isolate the issue: 
    lepton-netlist -g spice-sdb -o test.cir schematic.sch
ngspice -b -r test.raw test.cir
    Inspect test.cir for issues and review ngspice's stderr output.

Scheme server not connecting

Symptom: Cannot connect to lepton-schematic on localhost:9424. Cross-probe and back-annotation have no effect.

Solution:

  1. Verify the server is installed: 
    ls -la ~/.config/lepton-eda/scheme/autoload/pqwave-server.scm
    Or check $XDG_CONFIG_HOME if set.
  2. Verify the port is listening: 
    ss -tlnp | grep 9424
# Expected: LISTEN  127.0.0.1:9424
  3. If not installed, run the GUI install (File > Lepton Bridge > Watch Schematic) or headless: 
    api.lepton_config("install_server")
  4. Restart lepton-schematic after installation. The script is only loaded at startup via the autoload mechanism.
  5. Check for errors in lepton-schematic's terminal output. Start it from a terminal to see Scheme errors: 
    lepton-schematic schematic.sch

Cross-probe not working

Symptom: Probe Selected Net does nothing, or schematic-to-pqwave events are not received.

Solution:

  1. Verify bidirectional connectivity — test the TCP channel manually: 
    echo '$NET: "VCC"' | nc localhost 9424
    The VCC net should highlight in lepton-schematic if the server is running.
  2. Verify lepton-schematic is running with the server loaded. The server starts automatically when the script is loaded at startup.
  3. In pqwave, check the chat panel for error messages — connection errors are logged with the [lepton] prefix.
  4. If using the API, ensure the cross-probe client is connected: 
    # The GUI connects automatically when watching.
# For standalone API use, ensure connect:
from pqwave.bridge.lepton.cross_probe import LeptonCrossProbeClient
client = LeptonCrossProbeClient()
client.connect_to_server()
client.probe_net("VCC")

Port conflicts

Symptom: Cannot start the Scheme server because port 9424 is already in use.

Solution:

  1. Check what is using the port: 
    ss -tlnp | grep 9424
  2. If another instance of lepton-schematic is running, this is expected — only one instance can bind the port. Connect to the running instance.
  3. If a different application is using port 9424, change the port: 
    api.lepton_config("port", 9425)
    You must also edit pqwave-server.scm to match (change the pqwave-port variable at the top of the file), then reinstall it: 
    api.lepton_config("install_server")

DC annotations show wrong voltages

Symptom: The [DC:... V] stamps show incorrect values.

Solution:

  1. Ensure the simulation includes DC analysis. For transient-only simulations, pqwave uses the mean of the entire waveform, which may not represent the true DC operating point if the signal is not symmetric around its DC level.
  2. Add a .op analysis to your schematic to get an explicit operating point: 
    .op
.tran 0.1u 5m
  3. Manually pass explicit voltages for known nets: 
    api.lepton_annotate_dc({"VCC": 12.0, "out": 6.0})

File watcher not detecting saves

Symptom: Saving the schematic in lepton-schematic does not trigger re-simulation in pqwave.

Solution:

  1. The file watcher uses 1-second mtime polling. Wait at least 1 second after saving for the poll to detect the change.
  2. Verify the watched path is correct — the control bar shows the basename of the watched file.
  3. If the file is on a network filesystem, mtime may not update reliably. Use a local filesystem.
  4. Manually trigger simulation with the Simulate Now button in the control bar as a workaround.
  5. Check the chat panel for [lepton] messages — file change and pipeline events are logged there.

ngspice not found

Symptom: FileNotFoundError: ngspice not found.

Solution:

  1. Install ngspice: 
    # Debian/Ubuntu
sudo apt install ngspice
  2. Set the path in Settings: Settings > External Converter Paths > ngspice.
